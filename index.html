Incremental Attributes for MI

Extraction Progress and Partial Success support for MI Connectors via DCF
Created by @Rupa Singh (rupa.singh@salesforce.com) | June 15th, 2026
View Original Google Doc

📋 Executive Summary

Marketing Intelligence (MI) requires its connectors to report extraction progress at a granular, partitioned level (by time window or account) to facilitate large-scale onboarding. This document proposes leveraging DCF's existing Incremental Extract Attributes mechanism to support partial success reporting and recovery.

Key Features

✓ Structured Connector Support ✓ Opt-In Configuration ✓ New DataProvider API ✓ UI Integration ✓ Fast Retry Recovery

Dependencies

  • TD-0323935 - DCF Team Dependency
  • TD-0323936 - Data Services Team Dependency
💡 Value Proposition Eliminates the need for MI team to build a complex, slow, and brittle core-side orchestration layer for scheduling per-partition extracts. Successfully processed data remains preserved during mid-run failures, allowing retries to skip completed partitions.

📝 Requirements

In Scope

  • Granular progress reporting by time window or account
  • Partial success support for structured connectors
  • UI display of partial ingestion details
  • Fast recovery path with partition skipping

Out of Scope

⚠️ Not Included
  • Partial ingestion support for Structured JDBC connectors via DCF
  • UI visibility for non-incremental extract metadata
  • UI enhancements for user interactions and selective incremental ingestion

🔧 Technical Proposal

1. Connector Opt-In Configuration

Structured connectors opt into partial success functionality via feature configuration in their connector definition JSON:

"featuresConfig" : {
    "IncrementalUpdate" : {
        "deleteSupport" : "true",
        "partialSuccessSupport" : "true"
    }
}
✓ Backward Compatible Existing connectors continue to work unchanged. Only connectors explicitly opting in will get partial success support.

2. New DataProvider API Method

A new default method is added to the DataProvider interface:

public interface DataProvider<T>
{
    ...

    /**
     * Returns partial success details for structured connectors.
     * This information is persisted when extraction succeeds but
     * some partitions failed.
     *
     * @return JSON string with partition-level details (max 5000 chars)
     * @throws ConnectorException if unable to generate details
     */
    default String getPartialSuccessDetails() throws ConnectorException {
        return StringUtil.EMPTY;
    }

    ...
}

Return Value Format

Connectors can return any JSON format. Example:

{
  "runId": "abc-123",
  "schemaVersion": 1,
  "startedAt": "2026-05-07T08:00:00Z",
  "completedAt": "2026-05-07T08:42:11Z",
  "summary": {
    "total": 52,
    "succeeded": 47,
    "failed": 5
  },
  "partitions": [
    {
      "key": "account=123/window=2025-W14",
      "status": "SUCCESS",
      "rowsExtracted": 18432
    },
    {
      "key": "account=123/window=2025-W31",
      "status": "FAILED",
      "errorCode": "RATE_LIMIT"
    }
  ]
}

3. DCF Persistence Layer

DCF saves the partial success details as part of the data stream job for structured connectors that have opted in:

if (connector.hasOptedIntoPartialSuccessSupport()
       && !StringUtils.EMPTY.equals(dataProvider.getPartialSuccessDetails())) {

    DATA_STREAM_JOB_STATUS_REASON_OVERRIDE = "DATA_INGEST_SUCCESSFUL_WITH_ERRORS";

    // Truncated to a maximum of 5,000 characters
    DATA_STREAM_JOB_STATUS_DETAIL_OVERRIDE =
            truncate(dataProvider.getPartialSuccessDetails(), 5000);
}
⚠️ Size Limit The status detail is capped at 5,000 characters. Connectors must provide concise, actionable information within this limit. A proposal to increase this to 10,000 characters is under consideration.

Database Storage

Upon job completion, these details are permanently recorded as part of the job_metadata in the data_stream_job table, then copied to status_detail in data_stream_job_stat.

4. UI Integration

The UI displays an info icon (ⓘ) for successful jobs with warnings:

Status Icon Condition User Action
FAILURE ❌ Red error icon Job failed completely Click to view error details
SUCCESS (with warnings) ⓘ Blue info icon statusReason = "DATA_INGEST_SUCCESSFUL_WITH_ERRORS" Click to view partial failure details
SUCCESS ✓ No icon Job succeeded without issues -

Failure Flow

Two failure scenarios: extraction failure and ingestion failure

❌ Case A: Extraction Fails

If extraction fails, the process stops immediately. No ingestion is attempted, and incremental attributes are NOT saved.

❌ Case B: Ingestion Fails

If extraction succeeds but ingestion fails, the CDC event is published with failure status. The connector discards the job metadata, and incremental attributes are NOT saved.

🔄 INCREMENTAL_EXTRACT_ATTRIBUTES Lifecycle

Overview

jobMetadata is a Map<String, String> carried on DataStreamJob and persisted as a jsonb column in the data_stream_job table. It stores per-job execution context including connector name, data load mode, and incremental extract attributes.

INCREMENTAL_EXTRACT_ATTRIBUTES is a key within that map whose value is a JSON-serialized string of a Map<String, Object> — representing the connector plugin's cursor/watermark for incremental extraction.

Database Storage

Table: data_stream_job
Column: job_metadata (type: jsonb)

{
  "CONNECTOR_NAME": "SNOWFLAKE",
  "ISV_CONNECTOR_NAME": "",
  "DATA_LOAD_MODE": "INCREMENTAL",
  "s3JobRootPath": "tenant123/streams/ds-456/jobs/job-789",
  "INCREMENTAL_EXTRACT_ATTRIBUTES": "{\"lastModifiedDate\":\"2024-01-15T10:30:00Z\",\"cursor\":\"abc123\"}"
}
⚠️ String-Escaped JSON INCREMENTAL_EXTRACT_ATTRIBUTES is stored as a string within the JSON — a serialized nested JSON object, not an embedded JSON object. Connector code must properly serialize and deserialize this string.

Key Classes

Class Repo Role
DataStreamJob data-services JPA entity with @Column(name = "job_metadata", columnDefinition = "jsonb")
DataStreamJobService data-services saveInitialJob(), updateStreamJob() (merges metadata)
DataStreamEventsProcessor data-connectors Consumes ConnectorTriggerEvent from Kafka
DataStreamService data-connectors Orchestrates extraction, puts INCREMENTAL_EXTRACT_ATTRIBUTES into jobMetadata
CdcEventsListener data-connectors Consumes CDC event, extracts INCREMENTAL_EXTRACT_ATTRIBUTES, persists to data stream

15-Step Process Flow

# Service Action Detail
1 data-services saveInitialJob() Creates DataStreamJob row with initial job_metadata jsonb
2 data-services sendJobCreateEvent() Publishes ConnectorTriggerEvent to Kafka with jobMetadata
3 data-connectors Consumes Kafka event Deserializes ConnectorTriggerEvent
4 data-connectors buildProcessStreamJobSnapshot() Builds snapshot with jobMetadata from payload
5 data-connectors buildProcessStreamJobDetails() Copies jobMetadata, reads overrides
6 data-connectors updateDataStreamJobWithUpdateMetadataInSnapshot() Adds CONNECTOR_NAME, DATA_LOAD_MODE, etc. to jobMetadata
7 data-services updateStreamJob() Merges incoming map into existing job_metadata jsonb
8 data-connectors Extraction completes Gets incremental attributes from connector plugin
9 data-connectors convertIncrAttrToString() Serializes map to JSON string via ObjectMapper
10 data-connectors Put to jobMetadata map Adds INCREMENTAL_EXTRACT_ATTRIBUTES key
11 data-connectors dataJobServicesClient.updateJob() Persists full jobMetadata including attributes
12 data-services Job reaches terminal state Calls publishDataStreamStat()

📦 Partial Ingest for Unstructured Connectors

This section documents the existing partial ingest flow for unstructured connectors, which serves as the foundation for the structured connector implementation.

Two-Source Flow

Partial ingest details are saved to the data_stream_job_stat table through two sources that are merged:

Source 1: Spark Job
Produces row-level stats (rowsDroppedPartial, rowsDroppedErrorsInfo)
Source 2: data-connectors Extraction
Produces status overrides (statusDetail, statusReason)

Override Merge Logic

DataConnectorsStreamProvisioner.jobStatsOverride() conditionally overwrites Spark stats:

if (dataStreamJob.getStatus().equals(SUCCESS)) {
    String statusDetail = dataStreamJob.getJobMetadata()
        .get(Constants.DATA_STREAM_JOB_STATUS_DETAIL_OVERRIDE);
    String statusReason = dataStreamJob.getJobMetadata()
        .get(Constants.DATA_STREAM_JOB_STATUS_REASON_OVERRIDE);

    if (StringUtils.isNotBlank(statusDetail)) {
        dataStreamJobStat.setStatusDetail(statusDetail);
    }
    if (StringUtils.isNotBlank(statusReason)) {
        dataStreamJobStat.setStatusReason(statusReason);
    }
}

Conditions for Override

  • ✅ Job status is SUCCESS
  • ✅ Override values are non-blank
  • ✅ Only applies to DataConnectors location types

Column Length Constraints

Field Max Length DB Column
statusDetail 5000 chars status_detail
statusReason 1024 chars status_reason
rowsDroppedErrorsInfo 1024 chars rows_dropped_errors_info

REST Endpoint

GET /streams/{id}/status/{statusId}/logs

Returns only statusDetail (up to 5000 chars). The statusReason is NOT included in this endpoint's response.

🗄️ Database Changes (Proposed)

⚠️ Proposal Under Consideration This section documents a proposal to increase the status_detail column from 5000 to 10000 characters to accommodate full Spark stack traces.

Context

The current 5000-character limit truncates nested Java/Spark exception chains, often cutting off the root cause. Increasing to 10,000 characters would accommodate most full stack traces.

Affected Tables

Table Migration Current Proposed
data.data_stream_job_stat V022 VARCHAR(5000) VARCHAR(10000)
data.data_stream_job_stat_recent V097 VARCHAR(5000) VARCHAR(10000)
data.data_stream_interactive_job V098 VARCHAR(5000) VARCHAR(10000)

Migration SQL

File: V114__increase_status_detail_column_length.sql

-- Increase status_detail column from VARCHAR(5000) to VARCHAR(10000)
-- This is a metadata-only operation in PostgreSQL (no table rewrite, no lock).

ALTER TABLE data.data_stream_job_stat
    ALTER COLUMN status_detail TYPE VARCHAR(10000);

ALTER TABLE data.data_stream_job_stat_recent
    ALTER COLUMN status_detail TYPE VARCHAR(10000);

ALTER TABLE data.data_stream_interactive_job
    ALTER COLUMN status_detail TYPE VARCHAR(10000);
🚨 Critical: Deploy Order Matters
  1. First: Deploy admin-service-init migration (column widening)
  2. Then: Deploy data-services code (truncation constant updates)

If data-services deploys first but migration hasn't run, PostgreSQL will reject writes > 5000 chars.
